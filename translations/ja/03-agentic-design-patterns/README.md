# AIエージェント設計原則

## はじめに

AIエージェントシステムを構築する方法にはさまざまな考え方があります。生成AI設計では曖昧さがバグではなく特徴であるため、エンジニアがどこから手をつければよいか悩むことも少なくありません。そこで、開発者が顧客中心のエージェントシステムを構築してビジネスニーズを解決できるようにするために、人間中心のUX設計原則を作成しました。これらの設計原則は、厳密なアーキテクチャを指示するものではなく、エージェント体験を定義し構築していくチームの出発点となるものです。

一般的に、エージェントは次のような役割を果たすべきです：

- 人間の能力を広げ、スケールさせる（ブレインストーミング、問題解決、自動化など）
- 知識のギャップを埋める（知識分野を把握する、翻訳するなど）
- 個々人が他者と協働する際の好みに応じてコラボレーションを促進しサポートする
- 私たちをより良い自分にする（例：ライフコーチ/タスクマスター、感情調整やマインドフルネススキルの習得を支援する、レジリエンスを育むなど）

### エージェント（空間）

これはエージェントが動作する環境です。これらの原則は、物理的およびデジタルの世界でエージェントを設計する方法を示します。

- **つなぐこと、壊すことではなく** – 人々、イベント、そして行動可能な知識をつなげ、コラボレーションとつながりを可能にする。
  - エージェントはイベント、知識、人々をつなげる手助けをする。
  - エージェントは人々をより近づける存在であり、人を置き換えたり、軽視したりするものではない。
- **簡単にアクセス可能でありながら時には見えない存在** – エージェントは主にバックグラウンドで動作し、関連性があり適切な場合にのみ私たちに通知を行う。
  - エージェントは、認可されたユーザーがどのデバイスやプラットフォームでも簡単に発見しアクセスできる。
  - エージェントは、音声、テキスト、音などのマルチモーダルな入力と出力をサポートする。
  - エージェントは、ユーザーのニーズに応じて、前景と背景、プロアクティブとリアクティブをシームレスに切り替えられる。
  - エージェントは目に見えない形で動作する場合があるが、そのバックグラウンドプロセスや他のエージェントとのコラボレーションはユーザーに対して透明であり、制御可能である。

### エージェント（時間）

これはエージェントが時間を通じてどのように動作するかを示します。これらの原則は、過去、現在、未来を横断するエージェントの設計方法を示します。

- **過去**：状態とコンテキストを含む履歴を振り返る。
  - エージェントは、イベント、人、または状態だけでなく、より豊かな履歴データの分析に基づいて、より関連性の高い結果を提供する。
  - エージェントは過去のイベントからつながりを生み出し、記憶を活用して現在の状況に対応する。
- **現在**：通知ではなく、行動を促す。
  - エージェントは、人々との対話に包括的なアプローチを取り入れる。イベントが発生したとき、エージェントは静的な通知や形式にとどまらず、フローを簡素化したり、ユーザーの注意を適切なタイミングで引き付けるためのキューを動的に生成したりする。
  - エージェントは、コンテキスト環境、社会的および文化的な変化、そしてユーザーの意図に合わせて情報を提供する。
  - エージェントの対話は段階的であり、長期的にユーザーを支援するために進化し成長する。
- **未来**：適応し進化する。
  - エージェントは、さまざまなデバイス、プラットフォーム、モダリティに適応する。
  - エージェントは、ユーザーの行動、アクセシビリティのニーズに適応し、自由にカスタマイズ可能である。
  - エージェントは、継続的なユーザーとの対話を通じて形作られ、進化する。

### エージェント（コア）

これらはエージェント設計の中核となる重要な要素です。

- **不確実性を受け入れつつ信頼を確立する**。
  - エージェントに一定の不確実性があることは予期される。不確実性はエージェント設計の重要な要素である。
  - 信頼と透明性はエージェント設計の基盤となる層である。
  - エージェントがオン/オフになるタイミングは人間がコントロールし、エージェントのステータスは常に明確に表示される。

## これらの原則を実装するためのガイドライン

上記の設計原則を使用する際は、以下のガイドラインに従ってください：

1. **透明性**: AIが関与していること、その動作方法（過去の行動を含む）、フィードバックを提供する方法やシステムを修正する方法をユーザーに知らせる。
2. **コントロール**: ユーザーがカスタマイズ、設定の指定、個人化を行い、システムやその属性（忘れる能力を含む）を制御できるようにする。
3. **一貫性**: デバイスやエンドポイント間で一貫したマルチモーダル体験を目指す。可能であれば、馴染みのあるUI/UX要素を使用し（例：音声対話のためのマイクアイコン）、顧客の認知負荷を可能な限り軽減する（例：簡潔な応答、視覚的補助、「詳細を学ぶ」コンテンツを目指す）。

## これらの原則とガイドラインを使用して旅行エージェントを設計する方法

旅行エージェントを設計する場合、以下のように設計原則とガイドラインを活用できます：

1. **透明性** – 旅行エージェントがAI対応のエージェントであることをユーザーに知らせる。開始方法に関する基本的な指示を提供する（例：「こんにちは」メッセージ、サンプルプロンプト）。これを製品ページに明確に記載する。ユーザーが過去に尋ねたプロンプトのリストを表示する。フィードバックを提供する方法（サムズアップ/ダウン、フィードバック送信ボタンなど）を明確にする。エージェントに使用制限やトピック制限がある場合、それを明確に伝える。
2. **コントロール** – システムプロンプトなどを使ってエージェントを作成した後に、ユーザーがエージェントをどのように変更できるかを明確にする。エージェントの詳細さ、文体、およびエージェントが話題にすべきでない内容をユーザーが選択できるようにする。関連するファイルやデータ、プロンプト、過去の会話を表示および削除できるようにする。
3. **一貫性** – プロンプトを共有、ファイルや写真を追加、誰かや何かをタグ付けするためのアイコンが標準的で認識しやすいものであることを確認する。ファイルのアップロード/共有を示すためにクリップアイコンを使用し、グラフィックのアップロードを示すために画像アイコンを使用する。
